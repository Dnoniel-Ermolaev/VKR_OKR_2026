# Demo Cases for Presentation

Этот каталог содержит готовые материалы для демонстрации проекта на презентации. Здесь лежат и реальные `.xlsx`-файлы для импорта через web UI, и текстовые сценарии с ожидаемыми результатами.

## Что лежит в каталоге

- `generate_demo_cases.py`  
  UTF-8 генератор demo-`xlsx`. Если какой-то старый workbook открыт и заблокирован, скрипт сохраняет чистую копию с суффиксом `_fixed`.

- `structured_high_risk_demo_fixed.xlsx`  
  Короткий структурированный кейс высокого риска в исправленном UTF-8 виде. Его лучше использовать для демонстрации вкладки `Пациент`, Excel-импорта и автоматической переоценки.

- `protocol_gap_demo.xlsx`  
  Намеренно неполный STEMI-кейс. Подходит для показа protocol-driven dashboard: высокий риск уже распознан, но completion по протоколу остается низким, а критические шаги помечаются как pending.

- `patient_card_stemi_case.xlsx`  
  Почти полный кейс, собранный по реальной карте пациента из `Итог.png`.

- `presentation_scenarios.json`  
  Компактные тестовые сценарии для доклада: быстрый `/api/assess`, `free-text` пример и пример плохого ввода.

- `patient_card_stemi_case.md`  
  Описание кейса по карте пациента, его клинической логики и ожидаемого поведения системы.

## Как демонстрировать через сайт

### Вариант 1. Быстрый triage на вкладке `Оценка ОКС`

Используй сценарии из `presentation_scenarios.json`:

- `structured_high_risk_assess`
- `free_text_demo`
- `data_quality_demo`

Что показывать:

1. Какие поля вводятся.
2. Какой `risk_level` и `triage_category` вернула система.
3. Как система реагирует на неполный ввод (`missing_fields`, `data_quality_issue`).

### Вариант 2. Полный клинический кейс на вкладке `Пациент`

1. Выбрать пациента и создать визит.
2. Создать новый кейс.
3. Загрузить один из Excel-файлов через вкладку `Excel`.
4. Показать:
   - обновление данных кейса;
   - автоматическую переоценку;
   - выбранный протокол;
   - completion и alerts;
   - оценки и отчет.

## Рекомендуемые сценарии показа

### `structured_high_risk_demo_fixed.xlsx`

Что демонстрирует:

- импорт базовых structured данных;
- быстрое распознавание высокого риска;
- протокол `STEMI`;
- наличие уже внесенных назначений и вмешательств.

Ожидаемо:

- `risk_level = high`
- `triage_category = high_risk_fast_track`
- `protocol = STEMI`
- `completion_percent = 70.6`
- `critical_pending = ["Серия тропонина", "Серия ЭКГ"]`

### `protocol_gap_demo.xlsx`

Что демонстрирует:

- даже при уже распознанном STEMI case dashboard умеет показывать пробелы в ведении;
- completion снижен из-за отсутствия серийного тропонина, ЭхоКГ, части лабораторий или терапии;
- это хороший пример, зачем нужен protocol-driven контроль, а не только одноразовый triage.

Ожидаемо:

- `risk_level = high`
- `triage_category = high_risk_fast_track`
- `protocol = STEMI`
- `completion_percent = 5.9`
- в `critical_pending` остаются серийный тропонин, серия ЭКГ, ЭхоКГ, коронарография, ЧКВ и обязательная терапия
- в `alerts` появляются пропущенные критические шаги и отсутствие антикоагулянта

### `patient_card_stemi_case.xlsx`

Что демонстрирует:

- сценарий, максимально близкий к реальной клинической карте пациента;
- богатый структурированный импорт: vitals, labs, studies, procedures, medications, diagnoses;
- работу case lifecycle и отчета на содержательном кейсе.

Ожидаемо:

- протокол `STEMI`
- высокий риск
- `high_risk_fast_track`
- `completion_percent = 100.0`
- `critical_pending = []`
- в alerts остается только сообщение о критически высоком тропонине, что клинически ожидаемо

## Замечание по ожидаемым результатам

Результат зависит от того, загружаются ли данные в чистый кейс и как именно настроены статусы назначений и процедур. Для презентации лучше использовать новый кейс без посторонних записей и после импорта запускать `Переоценить`, если это нужно явно показать на экране.
